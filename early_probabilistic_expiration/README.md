# Early Probabilistic Expiration (XFetch)

## Идея

Проект №6 — вероятностный пересчёт кэша **ДО** истечения TTL.

Обычный cache-aside ждёт, пока TTL полностью истечёт, и тогда все запросы видят `MISS` и бегут в БД (cache stampede). XFetch решает проблему **превентивно**: каждый запрос с вероятностью решает «не пора ли обновить кеш?». Чем ближе к концу TTL — тем выше шанс. В итоге один из запросов обновляет данные **заранее**, и stampede не происходит.

## Формула

```
should_recompute = now > expiry - delta × β × (-ln(random()))
```

- `delta` — время последнего DB read (тяжёлый запрос → пересчёт раньше)
- `β (beta)` — агрессивность (1.0 = стандарт, 5.0 = ранний, 0.5 = поздний)
- `-ln(random())` — экспоненциальное распределение (вероятность растёт к expiry)

## Что хранится в Redis

```
product:1       → {"id":1, "name":"MacBook Pro 16", ...}     ← данные (TTL=120s)
meta:product:1  → {"delta": 0.035, "expiry": 1713550920.0}   ← метаданные XFetch (TTL=180s)
```

## Отличие от mutex lock

| Аспект | Mutex Lock (проект №5) | XFetch (этот проект) |
|--------|----------------------|---------------------|
| Когда срабатывает | ПОСЛЕ истечения TTL (реактивно) | ДО истечения TTL (превентивно) |
| DB reads | Ровно 1 (гарантия) | 1–5 (вероятностно) |
| Blocked потоки | 499 ждут retry | 0 (все читают из кеша) |
| Latency | Повышена (retry 80ms × N) | Нулевая (кеш валиден) |
| Сложность | Высокая (lock, Lua, retry) | Низкая (одна формула) |

## 6 тестовых сценариев

1. **Обычный cache hit/miss** — базовая проверка
2. **XFetch метаданные** — delta и expiry хранятся корректно
3. **Stampede БЕЗ XFetch** — TTL истекает, все идут в БД
4. **XFetch предотвращает stampede** — пересчёт до истечения
5. **Влияние beta** — агрессивный vs консервативный
6. **Влияние delta** — тяжёлые vs лёгкие запросы

## Результаты

С XFetch кеш обновляется **до** истечения TTL. При 50 запросах в момент истечения:
- Без XFetch: 10–50 DB reads (stampede)
- С XFetch: кеш уже обновлён, 0 full misses

На тяжёлых запросах (delta=1–2с) XFetch начинает пересчёт за 2–20 секунд до expiry, обеспечивая запас для завершения DB read.
